## 一、项目概述

各位老师好，我是来自七（2）班的李歆辰。我的项目是《基于声音与行为分析的"喵语"情感识别与需求解读系统》。

简单来说，就是我想用AI技术，让我家的两只猫"开口说话"——通过分析它们的叫声和动作，判断它们是饿了、想玩了、还是生气了。

我家养了两只猫，一只叫"可乐"，一只叫"草草"。在与它们相处的过程中，我发现猫咪虽然不会说话，但会通过不同的叫声（如短促的喵、长长的嚎叫）和丰富的肢体动作（如尾巴竖起、飞机耳、蹭腿）来表达各种需求。然而，作为主人，我往往很难准确听懂它们的真实意图。所以我希望利用AI技术，做一个真正能"听懂"猫咪的智能助手。

---

## 二、研究思路与技术路线

本项目的核心逻辑是**多模态融合**——将复杂的"理解猫语"拆解为两个子问题：

- **"它在做什么？"** → 视觉/姿态识别（图片）
- **"它在怎么叫？"** → 听觉/声音分类（音频）

最终在应用中结合声音和图像的分析结果，给出综合判断。

技术路线为：**数据采集 → 人工标注 → AI训练 → 模型验证 → 成果集成**

## 三、前期工作回顾

开题以来，我主要完成了以下几方面工作：

### 1. 文献调研：找到科学依据

我在网上查找了相关研究，发现科学家确实找到了"喵语"的一些规律：
- **叫声时长**：猫咪在积极情绪（如吃饭、玩耍）时，叫声较短；在消极情绪（如被关进航空箱）时，叫声拖得很长。
- **音调高低**：积极情绪时音调偏高，消极时音调偏低。
- **音调走势**：如果一声"喵"是先升高再降低，呈"∧"形，通常是开心的表现。

这些规律为我后续的数据标注提供了理论基础。

### 2. 情绪状态定义：建立分类标准

通过持续观察两只猫咪在不同情境下的表现，我制定了《猫咪行为-声音标签表》，初步定义了**5种状态**：

| 编号 | 状态名称 | 图标 | 典型表现 |
| :--- | :--- | :--- | :--- |
| ① | 饥饿乞食 | 🍚 | 短促叫声、蹭碗、跟随主人、尾巴竖起 |
| ② | 警惕警告 | ⚠️ | 飞机耳、瞳孔放大、尾巴猛甩、低沉嘶吼 |
| ③ | 心情愉悦 | 😊 | 呼噜声、眯眼睛、缓慢眨眼、身体放松 |
| ④ | 无聊求玩 | 🎾 | 叼玩具找主人、扒门、跑酷、兴奋短叫 |
| ⑤ | 标记领地 | 🏠 | 蹭家具、抓挠沙发、尾巴竖起颤抖 |

> **设计说明**：最初我定义了"撒娇求关注"和"无聊求玩"两个状态，但观察发现这两个状态的表现非常相似，容易混淆。经过调整，最终用"标记领地"替换了"撒娇求关注"，使5种状态的区分度更高。

### 3. 数据采集：最难也是最费时的部分

这是我花费时间最多的环节。

#### 3.1 采集方法创新：视频标注 + 关键帧提取

在数据采集过程中，我发现一个重要问题：**单张图片很难准确标注猫咪的情绪状态**。因为很多情绪是通过动态行为序列表达的，比如尾巴是"缓慢摆动"还是"猛甩"，耳朵是"慢慢变平"还是"突然贴头"，这些在静态图片中无法区分。

因此，我采用了**"视频录制 → 关键帧提取"**的方法：

```
录制视频（10-30秒） → 观看视频准确判断情绪 → 截取最典型的关键帧 → 标注
```

**这样做的好处**：
- 视频能捕捉完整的行为序列，帮助我更准确地判断猫咪情绪
- 从视频中选取的关键帧更有代表性，标注质量更高
- 最终训练用的仍然是图片（因为EasyDL图片分类模型只接受图片输入）

**采集的场景包括5种情绪状态**，针对每只猫分别采集：

| :--- | :--- | :--- | :--- | :--- | :--- |
| 饥饿乞食 | 已采集 | 已采集 | 已采集 | 已采集 | 已截取 |
| 警惕警告 | 采集较少 | 采集较少 | 采集较少 | 采集较少 | 待补充 |
| 心情愉悦 | 已采集 | 已采集 | 已采集 | 已采集 | 已截取 |
| 无聊求玩 | 已采集 | 采集较少 | 已采集 | 采集较少 | 部分截取 |

**目前采集进度**：

- 图片/视频帧：已截取 **约120张** 典型姿态图片

#### 3.3 数据采集的标准化

为了保证数据质量，我制定了详细的数据采集规范：

- **视频录制标准**：1080P分辨率，10-30秒/段，多角度拍摄（正面、侧面、俯视、平视）
- **音频录制标准**：WAV格式，3-5秒/段，安静环境，无背景噪音
- **文件命名规范**：`[猫咪名]_[情绪英文]_[序号].jpg`（如 `cola_hungry_001.jpg`）
- **数据均衡原则**：可乐和草草的数据量大致相等，避免模型偏向某一只猫

我还编写了一份详细的《数据采集方案》，发给家人协同执行，明确了分工和三周的采集计划。

### 4. 交互平台搭建：用Trae实现"Vibe Coding"

#### 4.0 开发工具：Trae与Vibe Coding

在开发过程中，我使用了一款名为**Trae**的AI编程工具。Trae是一个AI原生的集成开发环境（IDE），它支持一种叫"Vibe Coding"（感觉编程）的开发方式。

传统的编程需要一行一行写代码，而Vibe Coding是**用自然语言描述需求，AI自动生成代码**。比如我告诉Trae："帮我做一个猫咪图片上传区域，支持拖拽上传，上传后显示预览图"，Trae就能自动生成对应的HTML、CSS和JavaScript代码。

这种开发方式让我能专注于**功能设计和逻辑思考**，而不是纠结于语法细节，大大提高了开发效率。当然，AI生成的代码也需要我理解和调试，这个过程也让我学到了很多前端开发的知识。

#### 4.2 多模态融合

这是系统的核心思路：**同时结合图片识别和音频识别的结果，给出综合判断**。

就像我们日常生活中理解猫咪——也是同时看它的动作和听它的叫声来判断的。只看图片或只听声音容易误判，结合起来会更准确。

系统会根据用户上传的内容自动调整：只上传图片就只看图片结果，只上传音频就只听音频结果，两者都上传则各占一半权重综合判断。

## 四、遇到的困难与解决方案

### 困难一：猫咪不配合拍摄

**问题描述：** 猫咪不会听指令。当我想要录制"警告"状态的视频时，它偏偏在睡觉；当我举着手机等它叫，它就是不开口。

**解决办法：**
- 我采用了"蹲守法"——把手机放在常出现场景的位置，随时准备抓拍
- 利用自然触发条件，比如：
  - 拿罐头时 → 录制"饥饿"场景
  - 拿出逗猫棒时 → 录制"玩耍"场景
  - 陌生人敲门时 → 有机会拍到"警惕"状态

### 困难二：声音数据质量参差不齐

**问题描述：** 有些录音里有环境噪音（电视声、家人说话声），导致AI可能学到错误特征。

**解决办法：**
- 我尽量选择安静时段录制（比如家人不在客厅的时候）
- 对于有明显杂音的片段，我进行了重新录制，或者在剪辑时尽量截取干净的部分
- 使用Audacity软件进行音频剪辑，裁剪成3-5秒的标准片段

### 困难三：单张图片难以准确标注情绪

**问题描述：** 在标注图片时，我发现同一张图片可能有多种解释——猫咪抬头叫，可能是饿了，也可能是想玩，单看一张图片很难判断。

**解决办法：**
- 改用"视频录制 → 关键帧提取"的方法
- 先观看完整视频，结合上下文准确判断情绪
- 再从视频中截取最能代表该情绪的关键帧进行标注
- 这样标注准确率大大提高

### 困难四：AI平台的使用需要摸索

**问题描述：** 刚开始用百度EasyDL时，不太清楚如何设置标签、如何上传批量数据。

**解决办法：**
- 我看了官方的教学视频和文档
- 先上传了少量数据做了一次"试跑"，确认流程没问题后，再大规模上传

### 困难五：两只猫的数据是否需要分别训练

**问题描述：** 一开始我纠结是否需要为可乐和草草分别训练两个模型。

**解决办法：**
经过分析，我决定使用**统一模型**——将两只猫的数据合并训练一个通用情绪识别模型。原因：
- 猫咪的情绪表达是生物本能，跨个体通用（所有猫飞机耳都表示警惕）
- 合并后数据量更大（250张 vs 125张×2），模型更准确
- 训练一次即可，还能识别"其他猫咪"的情绪
- 在应用中通过前端"猫咪选择器"区分，不靠AI识别身份

## 六、阶段收获

通过这段时间的研究，我有几点体会：

1. **科学研究真的很辛苦**——光是录猫咪叫声就花了几十个小时，还要反复筛选、剪辑。

2. **理论和实践差距很大**——文献里写的规律看起来很清晰，但实际采集时才发现，每只猫都有自己的"口音"和习惯。

3. **AI不是魔法**——它需要高质量的数据才能工作，数据的好坏直接决定了模型的效果。

4. **我真正理解了"数据为王"的含义**——以前觉得AI厉害是因为算法牛，自己做下来才发现，**没有干净、充足、标注准确的数据，再厉害的算法也跑不起来**。录了100段模糊的叫声，不如20段清晰的叫声管用；而且同一个场景要多角度、多时段采集，才能让模型学会真正的规律，而不是记住某一只猫的某个特定动作。这段经历让我对"数据是AI的燃料"这句话有了切身体会。

5. **多模态融合的思路很有价值**——单一模态（只看图片或只听声音）容易误判，结合两种模态能显著提升判断准确率。这和人类理解猫咪的方式一致——我们也是同时看动作和听叫声来判断的。

6. **Vibe Coding让编程变得不一样**——用Trae这样的AI编程工具，我可以用自然语言描述需求，AI帮我生成代码。这让编程的门槛降低了很多，我可以专注于功能设计和逻辑思考。但我也体会到，AI生成的代码必须自己理解，不能盲目照搬，调试和理解代码的过程同样重要。
